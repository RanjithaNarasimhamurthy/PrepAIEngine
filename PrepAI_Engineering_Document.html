1. What the System Was Designed to Do — and Why It Mattered

Every year, hundreds of thousands of software engineers prepare for technical interviews by reading Reddit posts — raw, unstructured, often noisy accounts of real interview experiences at Google, Amazon, Meta, and hundreds of other companies. The signal is genuinely valuable: people describe exactly which problems were asked, how many rounds they faced, whether they received an offer, and what they wish they had studied. But the format makes it nearly useless at scale. You cannot search across 10,000 posts. You cannot ask "what does Amazon consistently test for L4 roles?" You cannot generate a personalised study plan from raw anecdotes.

PrepAIEngine was designed to close this gap entirely. The goal was to transform unstructured Reddit interview data into a queryable, analytically useful, and personally actionable intelligence platform — one that could answer three questions with genuine grounding in real data:

1. What is actually being asked? → Structured extraction of companies, roles, rounds, topics, and questions.
2. What patterns matter most? → Analytics dashboard with topic frequency, company volume, and offer rates.
3. How should I prepare? → Personalised week-by-week roadmaps and a RAG-powered AI assistant.

This mattered because existing tools either scraped data without structuring it, or provided generic advice disconnected from actual interview patterns. The gap between "raw Reddit post" and "actionable prep insight" was entirely a data engineering and AI problem — and one worth solving properly.

3. Where I Used Existing Tools — and Why

Used PRAW — Python Reddit API Wrapper

Reddit's API involves OAuth complexity, rate limiting, and pagination quirks. PRAW abstracts all of this cleanly. The scraper's real value is in the keyword filtering, deduplication logic, and comment aggregation — not the HTTP layer. Writing a raw Reddit client would have added two days of work for zero differentiated value.

Used Ollama — Local LLM Runtime

Running LLaMA 3 (8B) locally via Ollama gave a fully offline, zero-cost LLM with no API rate limits during development. For a pipeline processing thousands of posts, hitting an external API would have introduced latency, cost, and rate limit failures. Ollama's interface is a single POST call — trivial to integrate, trivial to swap out for Groq or another provider later.

Used sentence-transformers (all-MiniLM-L6-v2)

This model produces 384-dimensional embeddings that are fast, compact, and accurate enough for semantic search over interview text. It runs on Apple MPS (Metal) out of the box. The alternative — fine-tuning a custom embedding model — would have required labelled interview data that didn't exist. Using a pre-trained general-purpose model with domain-specific retrieval context (FAISS + RAG) was the right tradeoff at this stage.

Used FAISS — Vector Search

FAISS is the standard for in-process vector search at small-to-medium scale. I didn't need a managed vector database (Pinecone, Weaviate, Qdrant) because the index fits entirely in memory, persistence is handled by serialising to disk, and there's no multi-tenant access pattern. IndexFlatIP with L2-normalised vectors gives exact cosine similarity with zero approximation error at this scale.

Used PostgreSQL with JSONB

Interview data has a semi-structured schema — most records have company and role, but rounds, OA details, and prep insights vary significantly. JSONB columns for flexible fields gave adaptability without sacrificing queryability. GIN-indexed TEXT[] arrays for topics and questions enabled fast = ANY() filter queries. A fully schemaless store would have lost the relational guarantees needed for accurate analytics aggregations.

Used Redis

LLM responses are expensive to generate (10–30 seconds each) and deterministic for the same question — caching them for an hour is an obvious win. Search results cached for 5 minutes handle repeated queries without hitting PostgreSQL or FAISS. TTL-based expiry is built into Redis natively. Writing a custom cache layer would have been redundant work.

4. Where I Built From Scratch — and Why

Built LLM Extraction Prompt and Post-Processing Pipeline

This was the core intellectual problem of the system. Getting LLaMA 3 to reliably return structured JSON from noisy Reddit text required careful prompt engineering: a system instruction explicitly forbidding hallucination, a strict JSON schema with typed fields and constrained enumerations, canonical topic names to prevent free-form variation, and instructions to map questions to known LeetCode problem names rather than vague descriptions.

Post-processing was equally important. The sanitisation layer extracts the first valid JSON object from surrounding prose, normalises offer_status to a closed enum, validates round_type against a known set, and normalises company and role names through a canonical map of 150+ entries. This could not be off-the-shelf — extraction quality determines the quality of everything downstream.

Built Regex Company Name Fallback

LLaMA 3 frequently missed company names even when clearly present in the post text, returning null rather than hallucinating. I built a secondary extraction pass: after LLM extraction, if company is empty, scan the raw text with word-boundary regex against the canonical company map (sorted by name length descending to prevent partial matches). This recovered company attribution for a significant fraction of posts that would otherwise have been marked unknown.

Built Noise Post Filter

Many Reddit posts matching interview keywords were not interview experiences — general prep questions, rant posts, meme posts. Including them polluted the database with records containing no company, role, topics, questions, or rounds. I built is_useful_record() which discards any extracted record with zero signal across all five key fields. This ran at extraction time and was also exposed as a --clean pipeline command to retroactively remove noise. Filtering at extraction time rather than at query time kept the database clean and analytics accurate.

Built Hybrid Search Layer

The search endpoint combines FAISS semantic search (for free-text queries) and PostgreSQL structured filters (for company/role/topic facets). Neither alone was sufficient — semantic search misses exact company/topic matches; structured filters miss semantic intent. The hybrid layer deduplicates results by PostgreSQL ID, prepends semantic matches, and fills the remainder with structured results. No off-the-shelf tool handles this specific combination with deduplication and ordering.

Built Data-Driven Roadmap Scoring Engine

Rather than asking the LLM to generate a study plan from scratch (which produces generic, hallucinated output), I built a scoring engine first. It counts topic frequency across interviews matching the target company and role, applies base importance weights, and applies experience-level multipliers (beginners get foundational topics boosted; advanced candidates get System Design boosted). Topics are assigned to weeks in weighted order. The LLM is used only for a final 2–3 sentence narrative paragraph — where generation adds value because it is additive rather than structural. This approach grounds the roadmap in real data rather than model priors.

5. Tradeoffs Made

Exact FAISS search over approximate
Used IndexFlatIP (exact) rather than IndexIVFFlat (approximate). At 69–5000 vectors, exact search runs in milliseconds with no accuracy loss. At 100K+ vectors, approximate indexing becomes necessary. Correct choice for current scale with a clear migration path.
Local LLM over API
LLaMA 3 8B locally is slower (~10–15s per post) and less capable than GPT-4 or Claude. The tradeoff was cost, privacy, and rate limits — running locally costs nothing, has no rate limits, and doesn't send user data to third parties. For production with quality SLAs, swapping to Groq (LLaMA 3 via API at 500+ tokens/sec, free tier) would be the first change.
Single FAISS index over per-company indices
All interviews share one index. This simplifies the architecture but means semantic search is global — a query for "Google system design" might surface Amazon results if the text is similar. The structured filter layer compensates, but a more sophisticated system might maintain per-company indices or use FAISS metadata filtering.
Batch extraction over streaming
The pipeline processes posts in discrete batches rather than continuously. This introduces a lag between scraping and availability. A production system would use a message queue (Kafka, RabbitMQ) to stream posts through the pipeline in real time. Batch processing was chosen for simplicity — one Python script versus a distributed system.
No authentication layer
The API has no auth. Any client can call /ask or /roadmap. Acceptable for local development; not acceptable for public deployment. An API key header or JWT middleware would be added before hosting.

6. What I Would Do Differently in Hindsight

Initialise the database before running the pipeline
init_db() was wired into the FastAPI startup lifespan, so it only ran when the server started. Running the pipeline first caused "relation does not exist" errors and silent data loss to JSON. The fix was trivial, but the right design is to call init_db() explicitly at the top of pipeline.py — not rely on an implicit ordering dependency with the server.
Pin Python 3.12 as the explicit baseline from day one
The pydantic-core build failure on Python 3.14 was predictable — Rust-compiled packages lag behind new CPython releases by months. Documenting this constraint upfront would have prevented the environment rebuild.
Abstract the LLM call behind an interface from the start
Designing an LLMClient interface with a generate(prompt) -> str method rather than calling Ollama directly in two separate files would have made provider switching a one-line config change instead of a multi-file edit.
Track extraction quality metrics from the first run
I logged extraction progress but not quality — what fraction of records had company filled, how many were recovered by regex fallback, what fraction were filtered as noise. A post-extraction quality report would have surfaced data quality problems immediately rather than when inspecting the frontend.
Start with 200+ extracted records before evaluating search quality
Building the FAISS index with 20 records was fine for pipeline validation but gave an unrealistic picture of semantic search quality. A larger initial extraction would have produced more honest early results and surfaced retrieval quality issues sooner.
Summary

PrepAIEngine is a full-stack AI system that turns unstructured Reddit interview posts into a searchable, analytically rich, and personally useful interview intelligence platform. The core engineering challenge was the extraction pipeline — getting a local LLM to reliably produce structured output from noisy text, and building the fallback and filtering layers that make the extracted data trustworthy enough to power search, analytics, and recommendations. Every architectural decision — FAISS over a vector database, PostgreSQL with JSONB over a document store, local LLM over an API, hybrid search over pure semantic retrieval — was made with explicit reasoning about current scale, future migration paths, and the specific failure modes of each alternative. The decisions I would revisit are operational rather than architectural, which I take as a reasonable signal that the core design held up under real implementation pressure.
